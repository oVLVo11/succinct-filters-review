### A. 严格阶段变量（Day 3 重写，避免 `⌈log n⌉` 歧义）

为消除原文“同一字母 `i` 两用”，本笔记固定：

```text
i★(n)  := ⌈log n⌉          // 仅用于 Insert/Lookup 控制流（Claim 13 算法句）
epoch(n) := ⌊log n⌋         // 仅用于叙述“落在哪个二次幂区间”
```

事实：

- 对所有 `n ≥ 1`，`n ∈ [2^{epoch(n)}, 2^{epoch(n)+1})`。
- 当 `n` 恰为幂次 `2^k` 时，`i★(n) = epoch(n) = k`。
- 当 `n ∈ (2^k, 2^{k+1})` 时，`i★(n) = k+1`，`epoch(n) = k`。

**操作不变式（按算法句，已核实）**

设 `i = i★(n)`。一次 Insert/Lookup：

1. 新串尝试写入 `D_i`（若尚无更短前缀）；
2. 维护针对 `T_{i-1}, D_{i-1}` → `T_i, D_i`，并推进 `destroy` 旧结构与 `initialize(T_{i+1}, D_{i+1})`；
3. Lookup 检查 `D_{i-1}, D_i, T_{i-1}, T_i`。

**存储覆盖不变式（意图，Q1）**

无 failure 时，任一已接受前缀都存放在某个尚未销毁的 `T_j` 或 `D_j` 中，且 Lookup 用 `i★(n)` 检查的四个结构覆盖所有仍可能存有数据的层。

**对原文半开区间句的处理**

Claim 13 证明写：“`n ∈ [2^i, 2^{i+1})` 时数据在 `D_{i-1},T_{i-1},D_i,T_i`”。若此处 `i` 取 `epoch(n)`，则与 `n=3`（`epoch=1`）时算法实际写入 `D_2`（`i★=2`）**冲突**。  
**临时裁决（与 A 审阅一致）**：教学图与流程按 `i★` 绘制；正式归纳证明在统一变量前不关闭。见 `discussions/questions.md` Q3（阶段下标）与 Issue。

## 组件 7：长串 / 短串迁移去向

| 字段 | 内容 |
|---|---|
| 输入/输出 | 入：从 `D_{i-1}` 或 `T_{i-1}` `decrement` 出的串 `y`；出：写入 `D_i` 或 `T_i` |
| 保存信息 | 同组件 5 |
| 解决的问题 | 短串用真值表（`O(2^i)=O(n)` 量级）便宜处理；长串留在 succinct `D` |
| 缺失后果 | 短串塞进 `D` 浪费；或长串截断破坏正确性 |
| 依赖 | Claim 13 维护步骤 1–2 |
| 原文位置 | Claim 13 |
| 理解状态 | **已核实** |

规则：

```text
T_{i-1} → y  ⇒  insert(T_i, y∘0) 与 insert(T_i, y∘1)
D_{i-1} → y  ⇒  |y| > i → D_i ;  否则 → T_i
```

新插入的第 `n` 个前缀（长度 `ℓ_i`，通常 `≫ i`）默认进 `D_i`，不进 `T_i`。

### B. 常数 10 的配平（Day 3）

**原文事实（已核实）**：每次插入后执行维护过程 **10** 次。

**渐近机制（Q1，与 A 一致）**

在控制参数 `i = i★(n)` 下，当 `n` 从约 `2^{i-1}` 增到约 `2^i` 时，阶段内插入次数为 `Θ(2^i)`。  
黑盒承诺：

- `initialize(D)` / `destroy(D)`：各需 `O(m)` 次调用，`m` 为该实例容量（如 `2^{i-1}` 或 `2^{i+1}` 量级）；
- `decrement`：至多 `m` 次空调用 + 至多“当前存量”次成功删除。

因此“每次插入做常数 `c` 轮维护”在整个阶段提供 `Θ(c · 2^i)` 次底层调用，足以覆盖 `O(2^i)` 量级的 init/destroy/迁移工作 —— **对某个足够大的常数 `c` 渐近成立**。

**字面 `c=10`（Q3）**

接口只写 `O(m)`，隐藏常数未在 Claim 13 前展开，**无法从原文黑盒推出“恰好 10 一定够”**。验收：要么在附录伪代码中读出隐藏常数并复算，要么 Review 写“算法采用常数 10；渐近上常数轮足够，字面配平从略”。见 `discussions/issues/issue-constant-10.md`。

### C. 底层 `D(m,ℓ)` 接口（Claim 13 之前）

参数：容量 `m < u`，串长上界 `ℓ ≥ log m`。另有输入无关 `u^c` 预计算表。

| 原语 | 语义（原文） |
|---|---|
| `initialize(D)` / `destroy(D)` | 各连续调用 `O(m)` 次后完成；成功初始化后占 `O(m)` 位 |
| `insert(D,x)` | 要求 `log m < \|x\| ≤ ℓ`；插入后空间 ≤ `n(ℓ − log m + 2 log log log u) + O(m)`；可能 failure |
| `query(D,x)` | 返回是否存在 `D` 中某串为 `x` 的前缀；无 failure 时正确 |
| `decrement(D)` | 尝试删除任意一串并返回之；空调用总数 ≤ `m`；成功删除释放 `ℓ − log m` 位 |

**Core set / 长串 prefix matching**

- Claim 13 证明：写入某 `D_i` 的串来自 core set `B_{(i, ℓ_i]}`（库中无更短前缀覆盖），故对该 `D_i` 的插入序列服从文中分布 `D^{(i,ℓ_i]}_{c1 log u}`，failure ≤ `u^{-2C}`。
- “自动解决长串 prefix matching”的**接口含义**：`query(D,x)` 直接回答“是否存在已存长串为 `x` 的前缀”，无需调用方再扫多个长度。
- **内部实现（§5，Day 3 只到命名）**：main table（按 `st(x)`）→ subtable → adaptive prefixes + navigator → data blocks（动态块 + 后台重组为静态）。为何“长串集合上的 membership 能当 prefix matching 用”的位级理由：查询走指纹导航到至多一个相关块再解码比对；细节与 data block 冗余 `O(log log log u)` 仍为 **Q2**，Day 4 攻关。
