## 0. 这份笔记要解决什么问题

Day 1 已经知道论文研究“集合最终大小未知时的 filter”。Day 2 的目标是把这句话说准确：

- 数据结构维护的对象到底是什么？

- “unknown size”未知的是什么？
- `epsilon`、`n` 和 `u` 各自表示什么？

下面先给直观解释，再给接近论文原文的正式表述。全文默认 `log` 以 2 为底，原文第 2 页脚注 3 明确说明了这一点。

## 1. 基本对象

### 1.1 全集（universe）`[u]`

论文把所有可能出现的 key 组成的集合写成：

```text
[u] = {0, 1, 2, ..., u-1}
```

这里的 `u` 是全集大小。例如把学号编码成 `0` 到 `999999`，那么可以把 `u` 看成 `1000000`。

需要注意：

- `u` 不是当前已经插入的元素数；
- 本文的 “unknown sizes” 也不是说 `u` 不知道；
- `u` 会影响机器字长、哈希函数和失败概率等参数。

原文位置：第 1 页，第 1 节脚注 1。

### 1.2 当前集合 `S` 与当前大小 `n`

数据结构在某一时刻已经接收了一批 key，把这些 key 组成的逻辑集合记为：

```text
S ⊆ [u]
n = |S|
```

`n` 是当前集合大小，会随着插入而增长。论文只讨论插入和查询，没有把删除列为主定理支持的操作。

通俗地说：

- `u` 是“可能出现多少种 key”；
- `n` 是“现在实际存了多少个 key”。

论文的构造按“第几次插入”划分阶段。为避免混淆，本 review 在讲主结果时采用论文的插入序列模型；如果实际系统允许重复插入同一个 key，还需要额外说明如何去重，不能未经证明就把“插入次数”和“不同 key 的数量”完全等同。

原文位置：摘要、第 1 节和第 3.1 节。

### 1.3 容量上界 `N`

传统数据结构初始化时常要求提供容量 `N`，即预计最多同时存多少个 key。结构会按 `N` 分配空间。

如果实际只存了 `n` 个 key，而且 `n` 远小于 `N`，按 `N` 预留的空间就会浪费。本文希望空间跟随当前 `n`，而不是一直跟随事先估计的 `N`。

## 2. 两种操作

### 2.1 `insert(x)`

```text
insert(x): 把 key x 插入当前集合。
```

虽然 filter 为了省空间，不一定保存完整的 `x`，但从使用者角度看，`x` 已经进入逻辑集合 `S`。之后查询这个 `x` 时，filter 必须回答 YES。

### 2.2 `lookup(x)`

```text
lookup(x): 查询 x 是否属于当前集合。
```

对精确 dictionary，理想语义是：

- 若 `x ∈ S`，返回 YES；
- 若 `x ∉ S`，返回 NO。

对 filter，语义是：

- 若 `x ∈ S`，必须返回 YES；
- 若 `x ∉ S`，通常返回 NO，但允许以至多 `epsilon` 的概率返回 YES。

原文位置：第 1-2 页，第 1 节；第 3 页 Theorem 1 前后。

## 3. Dictionary 与 filter

### 3.1 Dictionary

Dictionary 给出精确答案，不允许误报或漏报。本文还讨论 key-value dictionary：

```text
insert(x, y): 插入 key-value 对 (x, y)
lookup(x): 若存在 (x, y)，返回 y；否则返回 NO
```

其中 `x ∈ [u]`，`y` 是一个 `v` 位的值。当 `v=0` 时，它退化为只判断 key 是否存在的精确 membership 结构。

原文位置：第 3 页，Theorem 1 与 Theorem 2 之间。

### 3.2 Filter

Filter 是 approximate membership data structure，即近似成员查询结构。它通过允许少量 false positive 换取更少空间。

| 实际情况 | 回答 | 名称 | 本文是否允许 |
|---|---|---|---|
| `x ∈ S` | NO | false negative（漏报） | 不允许 |
| `x ∉ S` | YES | false positive（误报） | 允许，但概率不超过 `epsilon` |

为什么不允许 false negative？因为 filter 至少要可靠地确认“插入过的元素可能存在”。如果连已经插入的 key 都可能回答 NO，那么它就不是本文研究的标准 approximate membership 模型。

## 4. 误报率 `epsilon`

`epsilon` 满足：

```text
0 < epsilon < 1
```

它表示：对一个没有插入的查询 key，filter 错误回答 YES 的概率上限。

例子：若 `epsilon = 0.01`，可以粗略理解为对非成员查询，误报概率最多为 1%。这不是说“每 100 次一定恰好错 1 次”，而是概率上界。

### 4.1 概率来自哪里

数据结构预先使用随机比特构造哈希函数等随机组件。插入序列本身可以是任意固定序列；概率是对这些预先生成的随机比特而言的。

Theorem 10 把两类概率分开：

1. **结构失败概率**：在任意插入序列上，结构曾报告 failure 的总概率至多为 `δ=u^{-C}`；
2. **查询误报概率**：在没有 failure 的条件下，每个 membership query 的 false positive rate 至多为 `epsilon`。

这两个概率不能混为一谈。`failure` 是数据结构内部的罕见坏事件，例如某个桶负载过高；false positive 则是 filter 模型本来允许的查询误差。

原文位置：第 9 页 Theorem 10 第 3、4 条。

## 5. 为什么没有 false negative

本文先用全局哈希函数 `h` 把 key 变成长二进制串。插入 `x` 时，结构保存 `h(x)` 的一个前缀；查询 `x` 时，检查数据库中是否存在某个字符串是 `h(x)` 的前缀。

如果 `x` 确实插入过，那么当时保存的就是 `h(x)` 自己的前缀。因此只要底层 prefix matching 结构没有报告 failure，查询一定能找到这个前缀并返回 YES。

这说明“无漏报”主要来自构造本身，而不是概率估计。概率分析主要用于控制：

- 非成员碰巧匹配某个已存前缀的概率；
- 底层随机结构报告 failure 的概率。

原文位置：第 9-10 页，第 3.1 节 Theorem 10 后。

## 6. “Unknown sizes”准确是什么意思

本文所说的 unknown sizes 是：

> 初始化数据结构时，没有给出最终集合大小，也没有一个足够紧的容量上界 `N`；结构仍要让占用空间随当前大小 `n` 增长。

它不表示：

- 全集大小 `u` 未知；
- 当前已经插入多少 key 完全不可得；
- 数据结构可以无限插入而不受 `n<u` 等条件约束。

### 6.1 为什么不能把 `N` 估得很大

当然可以给出一个非常保守的 `N`，但传统结构会长期按照 `N` 占用空间。如果真实 `n` 很小，这违背本文“空间依赖当前 `n`”的目标。

### 6.2 为什么 exact dictionary 比 filter 更容易扩容

Exact dictionary 保存了足够信息，可以枚举旧 key 并重建更大的结构。Filter 为省空间已经丢失一部分原始信息，通常无法从内部状态恢复所有旧 key。因此“建一个更大的表，再把旧 key 搬过去”并不能直接使用。

原文位置：第 2 页；第 4-5 页 §1.3 Previous Construction。

## 7. 本文中 “succinct” 的含义

“Succinct”不能只翻译成“空间很小”。原文给出的定义是：

> 使用的信息论最小空间，加上一个渐近更小的冗余项。

设理论上至少需要 `L` 位：

- 使用 `O(L)` 位，只说明量级相同，常数可能很大；
- 使用 `L + o(L)` 位，或者 `(1+o(1))L` 位，才体现主项的领先常数趋近 1。

本文尤其关心 unknown-size filter 中 `log log n` 项的领先常数。前作只做到 `O(log log n)`，而本文把上界写成明确的 `log log n`，从而在 `epsilon=o(1)` 时与下界的领先常数对齐。

原文位置：第 2 页脚注 2；摘要和第 2-3 页。

## 8. 已知容量时的经典空间基准

如果事先知道最多存 `N` 个 key，误报率为 `epsilon`，经典 filter 的信息论空间基准为：

```text
N log(1/epsilon) bits
```

Pagh、Pagh 和 Rao 的结构达到：

```text
(1+o(1)) N log(1/epsilon) bits
```

直觉上，`epsilon` 越小，要求越严格，每个 key 就需要更多位来减少随机碰撞。把 `epsilon` 缩小一半，大约需要再增加一位区分信息，因此主要成本是 `log(1/epsilon)`。

原文位置：第 2 页第 1 节；引用 [8, 23, 27]。

## 9. 未知规模为什么多出 `log log n`

### 9.1 构造层面的直觉

不知道最终容量时，数据结构要分多个增长阶段处理插入。阶段 `i` 大约处理规模 `2^{i-1}` 到 `2^i`。

如果每个阶段都可能产生误报，总误报率就要对所有阶段相加。论文给第 `i` 阶段设置更小的误报预算。为了把第 `i` 阶段误报压低，哈希前缀要多保存约 `log i` 位。

由于当前阶段编号大约是：

```text
i ≈ log n
```

所以：

```text
log i ≈ log log n
```

这解释了本文构造中 `log log n` 的来源。

### 9.2 不能把直觉当成下界证明

上面的阶段预算解释说明“本文构造为什么付出 `log log n`”，但还没有证明任何可能的 unknown-size filter 都必须付出这些位。真正的不可避免性来自 Pagh、Segev、Wieder 的信息论下界。原论文只引用该下界，没有在本文中重证。

因此，严谨表述应是：

- 构造直觉：多阶段误报预算导致每个 key 多存约 `log log n` 位；
- 理论结论：PSW 下界证明 unknown-size 模型确实必须支付近似 `log log n` 的额外空间。

## 10. 前缀长度公式与误报率

第 `i` 阶段插入 key `x` 时，论文保存 `h(x)` 的 `ell_i` 位前缀，其中：

```text
ell_i = i + log(1/epsilon) + log i + log log log u + 2        (公式 2)
```

各项的直观作用：

| 项 | 直观作用 |
|---|---|
| `i` | 抵消第 `i` 阶段最多约 `2^i` 个 key 带来的匹配机会 |
| `log(1/epsilon)` | 把总误报控制到目标 `epsilon` |
| `log i` | 给不同阶段分配可求和的误报预算 |
| `log log log u` | 提供额外余量，使对所有阶段的概率和严格小于 `epsilon` |
| `+2` | 常数安全余量 |

对一个非成员查询，第 `i` 阶段最多有约 `2^i` 个前缀，每个前缀匹配的概率为 `2^{-ell_i}`。用 union bound（并合界）把所有阶段相加：

```text
Σ 2^i · 2^{-ell_i}
= epsilon · Σ 1/(4i log log u)
< epsilon
```

这里使用了全局哈希函数至少具有 pairwise independence（两两独立性）。原文的哈希函数实际上具有更高的有限独立性。

原文位置：第 10 页，第 3.1 节公式 (2)。

## 11. 正式 Theorem 10 的完整保证

### 11.1 参数与模型

正式定理设：

```text
0 < epsilon < 1
δ = u^{-C}，其中 C > 1 可任意选定
机器字长 w = Θ(log u)
n = ω(log u) 且 n < u
```

`n=ω(log u)` 表示 `n` 比 `log u` 增长得更快，即 `n/log u → ∞`。这是渐近条件，不是说某个具体小数据集只要稍大于 `log u` 就自动满足所有结论。

### 11.2 空间

插入任意 `n` 个 key 后，数据结构主体使用：

```text
n(log(1/epsilon) + log log n + O(log log log u)) bits
```

此外还有：

```text
u^c bits
```

的预计算空间，其中 `0<c<1` 可以取任意小常数，而且这些预计算内容与输入无关。

不能在记录正式定理时漏掉这部分额外空间。摘要中的简洁公式主要强调随当前数据规模变化的主体空间。

### 11.3 时间

每次插入和查询的最坏情况时间都是：

```text
O(1)
```

这里不是“平均下来每次 `O(1)`”，而是每次操作本身都只执行常数步。扩容和迁移工作被拆散到连续插入中完成。

### 11.4 失败概率

对任意插入序列，数据结构在整个过程中曾报告 failure 的概率至多：

```text
δ = u^{-C}
```

概率取自预计算的随机比特。`C>1` 可以选得更大，从而让失败概率成为关于 `u` 的更高次多项式倒数。

### 11.5 查询正确性

在没有报告 failure 的条件下：

- 对成员没有 false negative；
- 对非成员的 false positive rate 至多为 `epsilon`。

原文位置：第 9 页 Theorem 10。

## 12. Theorem 1 与 Theorem 10 为什么看起来不同

论文给了两个层次的表述：

### 12.1 非正式 Theorem 1

当：

```text
n > u^0.001
```

空间写成：

```text
(1+o(1)) n(log(1/epsilon)+log log n)
```

并称插入和查询以高概率具有最坏情况常数时间。

### 12.2 正式 Theorem 10

条件写得更一般：

```text
n = ω(log u)，n < u
```

并把空间中的低阶项和输入无关的预计算空间完整写出：

```text
n(log(1/epsilon)+log log n+O(log log log u)) + u^c
```

### 12.3 二者如何衔接

当 `n > u^0.001` 时：

- `log log n = Θ(log log u)`；
- `log log log u = o(log log n)`；
- 因为 `c` 可取任意小正常数，可以选 `c<0.001`，使 `u^c=o(n)`。

因此，正式定理中的低阶项和预计算空间可以被吸收到 `(1+o(1))` 中，得到摘要和 Theorem 1 的简洁写法。

这说明 `n>u^0.001` 主要帮助论文把完整空间式整理成漂亮的 succinct 形式，但正式构造本身在更宽的 `n=ω(log u)` 范围内陈述。

## 13. `(1+o(1))` 到底是什么意思

若空间为：

```text
(1+o(1)) · n(log(1/epsilon)+log log n)
```

意思是：当参数逐渐变大时，实际空间与主表达式的比值趋近 1。

它不表示：

- 实际空间永远刚好等于主表达式；
- 小规模数据下额外开销一定很小；
- 可以忽略论文列出的所有参数条件。

更准确的理解是：论文不仅做到了同一个大 O 量级，还把主项前面的渐近常数压到了 1。

## 14. “最坏情况 O(1)、期望 O(1)、均摊 O(1)”的区别

| 保证 | 通俗解释 |
|---|---|
| expected `O(1)` | 对随机性取平均后，一次操作耗时为常数；个别运行可能很慢 |
| amortized `O(1)` | 一长串操作平均每次为常数；其中某一次可能耗时 `Θ(n)` |
| worst-case `O(1)` | 每一次操作本身都有常数时间上界 |
| worst-case `O(1)` with high probability | 除一个极小的随机失败事件外，整个结构中的每次操作都有最坏情况常数上界 |

本文的进步是：在空间随当前 `n` 保持 succinct 的同时，把前作的 expected amortized insertion 改进为带高概率保证的 worst-case constant time。

原文位置：第 2-4 页；第 9 页 Theorem 10。

## 15. Day 2 核查结论

### 已由本文原文确认

1. `unknown size` 指没有最终规模或紧容量上界，空间要依赖当前 `n`；
2. filter 不允许 false negative，允许概率至多为 `epsilon` 的 false positive；
3. 正式空间为 `n(log(1/epsilon)+log log n+O(log log log u)) + u^c`；
4. 正式失败概率为 `u^{-C}`，对任意插入序列中的“曾失败”事件成立；
5. 每次插入和查询都是 worst-case `O(1)`；
6. Theorem 1 与 Theorem 10 的参数写法不同，但可以通过低阶项分析衔接。

### 仍不能只凭本文确认

1. PSW 下界的完整证明过程；
2. 常数 `epsilon` 下 `log log n` 项是否能声称领先常数严格为 1；
3. 在实际实现中，预计算表和高独立性哈希的常数开销有多大；
4. 重复插入的工程语义如何处理。

这些内容应保留为待核查问题，不把直觉写成定理。
