## 1. 论文研究对象

这篇论文研究的是 membership data structure：维护一个来自全集 `[u]` 的动态集合 `S`，支持：

- `insert(x)`：把元素 `x` 插入集合；
- `lookup(x)`：判断 `x` 是否属于集合。

如果要求答案完全正确，这类结构叫 dictionary；如果允许对非成员元素以小概率误报 YES，但不允许对成员元素误报 NO，则叫 filter。Bloom filter 属于后一类。

本文关注的不是“能不能做 membership query”，而是更细的问题：当集合最终大小未知时，能不能让空间使用量始终接近当前集合大小 `n = |S|` 所需的信息论最优空间，同时仍然保持常数时间操作。

## 2. 初步理解

标准 filter 或 dictionary 通常在初始化时需要给定容量上界 `N`。结构会按照 `N` 分配空间，所以即使当前只插入了很少的元素，也会占用接近容纳 `N` 个元素的空间。如果 `N` 只是保守估计，空间浪费会更严重。

对 exact dictionary 来说，扩容相对自然：可以在容量满时新建更大的结构，并把旧结构中的真实键搬过去。问题在于 filter 本来就不保存完整元素，只保存足以回答近似成员查询的信息。因此，旧 filter 里可能无法恢复全部原始元素，不能像普通动态数组或哈希表那样直接搬迁。

这正是本文问题困难的核心：既想“不预知最终大小”，又想“空间按照当前 `n` 计算”，还想“误报率受控、无漏报、操作最坏情况常数时间”。

## 3. 论文主要结果初读

论文给出两个非正式主结果：

1. Dynamic filter：当当前元素数 `n > u^0.001` 时，支持误报率 `epsilon` 的近似成员结构使用
   `（1 + o(1)） n (log(1/epsilon) + log log n)` bits，
   并且插入和查询都以高概率达到最坏情况常数时间。

2. Dynamic dictionary：维护键值对时，空间为
   `（1 + o(1)） n (log(u/n) + v + O(log log log u))` bits，
   并且插入和查询都以高概率达到最坏情况常数时间。

我需要重点核查第一个结果中的空间项、下界、参数限制以及“高概率最坏情况常数时间”的含义。

## 4. 为什么不是简单扩容 Bloom filter

普通扩容思路是：容量不够时新建一个更大的 filter，然后把旧元素搬过去。这个思路在 Bloom filter/filter 场景里直接失败，因为旧 filter 不保存完整集合，只保存压缩后的近似信息。它能回答“某元素可能在集合里吗”，但通常不能列举出所有已插入元素。因此，旧元素无法可靠迁移到新结构。

另一种办法是保留多个不同阶段的 filter，新元素进入新 filter，查询时检查所有 filter。这样可以避免迁移，但会带来两个问题：

- 查询可能要检查 `O(log n)` 个结构，破坏常数时间目标；
- 为了让总误报率不超过 `epsilon`，各阶段需要分配误报预算，空间会出现额外 `log log n` 项。

本文的关键不是“会扩容”，而是在未知当前增长规模下，把空间中不可避免的额外 `log log n` 项做到领先常数最优，并同时保证操作最坏情况常数时间。

## 6. 总结
这篇论文不是简单设计一个可扩容 Bloom filter，因为 Bloom filter 的本质是用丢失信息的方式换取空间节省。扩容一个精确字典时，可以枚举旧结构中的元素并重新插入新结构；但 filter 通常不能从内部状态恢复已插入集合。所以，直接“建一个更大的再搬过去”不可行。保留多个旧 filter 虽然能避免搬迁，却会增加查询结构数量，并且为了控制总误报率，需要把误报预算分配给多个阶段，带来额外空间。本文真正解决的是：在集合最终大小未知的动态过程中，让空间始终接近当前 `n` 的最优界，同时把额外 `log log n` 项做到最优领先常数，并保持最坏情况常数时间操作。
