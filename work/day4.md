# Day 4 工作分配：证明闭环、复杂度核查与后续研究筛选

计划日期：Day 3 完成后执行  
角色不变：A 刘威；B 张书铖；C 陈戚。

## 一、与 Day 3 的衔接原则

Day 4 不重新讲一遍构造，而是把 Day 3 的组件笔记和依赖图变成可逐项检查的证明链。进入 Day 4 前，应至少具备：

- A 的引理依赖、空间分解和随机性假设笔记；
- B 的十个核心组件、阶段不变式、体系结构图；
- C 的经典文献矩阵和带核查状态的后续研究候选；
- 已编号的 Q1/Q2/Q3/Q4 及负责人。

若某项未完成，责任人先补齐并在会议记录中写明延期原因；其他成员可并行推进不依赖该项的工作，但不得把证明表中的空白写成肯定结论。

## 二、Day 4 共同目标

1. 对无 false negative、误报率、空间、查询时间、插入时间和高概率 failure 保证分别形成完整证明链；
2. 写出状态清楚、能与论文构造逐项对照的伪代码级实现说明；
3. 对 2020 年后候选论文做正文级筛选，避免把“题目相近”误写成“改进本文”；
4. 完成第二轮实质性交叉审阅，并为 Day 5 三个章节初稿准备可靠素材；
5. 保留争议、修订和 AI 人工核验的版本轨迹。

## 三、A 与 B 共同任务：证明表

新建 `notes/proof-table.md`。A 负责空间、概率和参数列，B 负责不变式、操作和实现列；两人分别提交，禁止由一人一次性代填。表格至少包含：

| 结论 | 精确命题 | 所需不变式/引理 | 证明步骤 | 原文位置 | 适用条件 | 当前缺口 | 复核人 |
|---|---|---|---|---|---|---|---|
| 无 false negative | 所有已插入 key 查询成功 | 前缀保存与迁移不变式 | 分阶段说明 key 始终可被查到 | 页码/引理 | 无 failure 等条件 | 待填 | C |
| 误报率不超过 `epsilon` | 非成员命中任一前缀的概率受控 | 前缀长度、逐阶段预算、union bound | 写出逐项求和 | 页码/公式 | 哈希/对手假设 | 待填 | C |
| 空间界 | 达到 Theorem 10 的位数 | dictionary 空间、层级求和、并存峰值 | 主项和低阶项分解 | 页码/定理 | `n,u,epsilon` 范围 | 待填 | C |
| 查询时间 | worst-case `O(1)` | 只检查四个活跃结构 | 接口调用数与单次成本 | 页码/Claim | 无 failure 等条件 | 待填 | A |
| 插入时间 | worst-case `O(1)` | 每次常数轮后台工作 | 阶段工作量配平 | 页码/Claim | 字典接口保证 | 待填 | A |
| 高概率保证 | 整段序列失败概率至多目标值 | 各 failure 事件及重建分析 | 对式 (3)(4)(5) 等逐项并合 | 页码/公式 | 随机性/序列条件 | 待填 | B/C |

每一行必须区分：论文明确给出的命题、小组补出的解释、尚未验证的推导。不能用教学示例替代一般性证明。

## 四、成员 A：刘威

### 1. 主任务：概率、空间和渐近条件核查

在 `notes/memberA/probability-and-space-proof.md` 中详细完成：

1. 逐阶段误报预算如何分配，前缀长度公式每一项如何进入概率界；
2. union bound 对哪些阶段、结构或失败事件使用，求和范围是什么；
3. false positive 与数据结构报告 failure 两类概率如何区分；
4. 式 (3)(4)(5) 等失败事件如何汇总到正式高概率保证；
5. 新旧结构并存、truth table、`D/T` 子结构和预计算空间如何计入峰值；
6. `O(log log log u)`、`u^c` 被主项吸收所需的参数范围；
7. Theorem 10 到 Theorem 1 的渐近化简；
8. worst-case `O(1)` 是否同时覆盖 lookup/insert，以及保证是否以“未报告 failure”为条件。

对 PSW 下界，只在已读原文足以支持时补充证明骨架；否则说明本文如何引用该下界及本 Review 应采用的谨慎表述。

### 2. 审阅职责

- 第一审阅 B 的伪代码和阶段边界，重点检查状态定义、循环不变式和单次工作上界；
- 第二核查 C 文献中声称“达到更优空间/时间”的条目是否使用相同模型和概率保证。

### 3. 提交节奏

```text
proof(A): derive false-positive budget and union bounds
proof(A): verify space peak and asymptotic conditions
review(A): audit pseudocode invariants and complexity claims
```

## 五、成员 B：张书铖

### 1. 主任务：伪代码级实现说明

新建 `notes/memberB/pseudocode.md`，至少包含：

```text
Initialize
Lookup(x)
Insert(x)
StartNewStage()
MigrateOneStep()
FinalizeStage()
HandleFailureOrRebuild()   // 仅在原文支持该接口时保留
```

伪代码无需运行，但必须先定义全部状态：当前 `n`、阶段编号、前缀长度、全局哈希、`D_{i-1},D_i,T_{i-1},T_i`、迁移游标、初始化/销毁进度、failure 标志和底层 dictionary 接口。

每个过程之后写四项说明：前置条件、修改的状态、保持的不变式、worst-case 工作量。若论文只证明抽象接口而非工程实现，应明确写“抽象伪代码”，不得虚构内存布局。

### 2. 正确性与时间证明

在 `notes/memberB/proof-notes.md` 中补齐：

- 插入前、迁移中和迁移后三种状态下为何不会漏报；
- 查询为何只访问常数个结构；
- 每次插入执行常数轮维护为何足够；
- destroy、initialize、decrement/迁移如何同时推进；
- 底层 dictionary 的 worst-case 接口保证如何传递到上层 filter；
- failure/rebuild 若存在，其触发条件和对公开 API 保证的影响。

### 3. 审阅职责

- 第一审阅 A 的概率/空间推导，检查其是否与具体结构及阶段索引一致；
- 回应 C 从“未读论文读者”视角提出的伪代码断点。

### 4. 提交节奏

```text
impl(B): define state and pseudocode for all operations
proof(B): verify migration invariants and worst-case work
review(B): align probability proof with stage construction
```

## 六、成员 C：陈戚

### 1. 主任务：后续研究正文级筛选

在 Day 3 候选列表基础上修订 `notes/memberC/post-2020-work.md`，并新建 `references/post-2020-matrix.md`。每个保留条目记录：

1. 是否正式发表、会议/期刊/预印本状态、DOI 或正式 PDF；
2. 是否引用本文，引用发生在背景、比较还是技术继承处；
3. 解决的模型是否同样 unknown size；
4. 是否改进空间、时间、删除、适应性或工程性能；
5. 保证属于 worst-case、expected、amortized 还是 high probability；
6. 结论来自摘要、正文、定理还是实验；
7. 与本文的关系判断及其原文证据；
8. 状态：纳入 Review、仅作旁支、排除、待核查。

至少对 3 个最相关候选阅读正文关键章节。数量不足时如实报告，不以应用论文或仅题目相似的论文填充。

### 2. 主任务：证明可读性与引用核查

- 以未读过原论文的读者身份复核 `notes/proof-table.md`；
- 对六个证明目标各提出至少一个“能否回原文验证”的问题；
- 检查所有文献元数据、引用位置和 BibTeX；
- 把后续研究与原论文已提及的经典工作分开，不制造时间线混乱；
- 初步形成“本文优点、限制、适用条件、后续方向”的证据清单，但暂不写成终稿评价。

### 3. 审阅职责

- 第一审阅 A/B 共同证明表：重点检查术语首次出现是否定义、每一步是否有原文位置、是否把条件遗漏；
- 复核 B 的伪代码能否由未读论文者跟随；
- 对 A/B 的理论判断只做证据核查，不替代其证明责任。

### 4. 提交节奏

```text
refs(C): verify post-2020 candidates from primary sources
review(C): audit proof table for traceability and readability
refs(C): classify evidence for strengths and limitations
```

## 七、共同交叉审阅与问题关闭

第二轮审阅按固定角色进行：

- A 审 B：伪代码、迁移不变式、最坏情况时间；
- B 审 A：概率预算、空间求和、参数条件；
- C 审 A+B：证明链可读性、原文定位和条件遗漏；
- A+B 共同核查 C：后续研究是否真与本文可比。

每人至少留下 3 条实质性评论。每条评论包含文件和小节、主张、缺失证据、建议修改、作者回应和关闭依据。优先通过 PR review 或 Issue 留痕；若使用 Markdown 记录，则新建 `discussions/review-day4.md`。

处理 `discussions/questions.md` 时：

- Q2、Q4 必须给出证据充分的结论，或明确升级为阻塞 Day 5 的问题；
- Q1 可在复核完成后转 Q0；
- Q3 若确属论文省略细节，应在 Review 中如实说明抽象接口边界，不擅自实现；

## 八、Day 4 会议与口头质询

新建 `discussions/meeting-day4.md`，三人轮流解释，另外两人必须追问：

1. 为什么不会漏报；
2. 为什么误报率不超过 `epsilon`；
3. 为什么扩容不会造成某次插入过慢；
4. 为什么查询只需常数次底层操作；
5. 为什么总空间接近信息论最优；
6. failure 概率和 false-positive 概率为何不是一回事；
7. 后续研究究竟改进了什么，是否处于同一模型。

记录中保留错误回答、质疑、修正和仍未解决项，不只记录最终结论。每人至少复述一个非本人负责的完整证明链，为后续口头检查做准备。
